# Task Plan — RBAC (역할 기반 접근 제어)

## 목표
현재 스택(FastAPI+SQLAlchemy) 유지 결정에 따라, v3 우선순위 1번인 RBAC을 구현한다.
현재는 `/api/auth/login`이 이메일만 확인하고 비밀번호/세션/권한 검사가 전혀 없는 상태 — 이를
"로그인(비밀번호 검증) + 세션 토큰 + 역할별 트랜잭션 실행 제한"으로 강화한다.

## 설계 결정
- **인증 방식**: 세션 테이블(DB) 기반. JWT/OAuth2는 구현 난이도·의존성 대비 프로토타입에 과함 —
  이 결정 자체를 README에 명시해 향후 실제 운영 전환 시 고려사항으로 남긴다.
- **비밀번호 해시**: 표준 라이브러리 `hashlib.pbkdf2_hmac`만 사용(신규 의존성 추가 없음).
- **인가 범위**: 상태를 변경하는 POST 엔드포인트만 역할로 게이팅한다. GET(조회)은 이번 범위에서
  인증을 강제하지 않는다(내부망 전제, 명시적 비범위 — 필요시 v4에서 확장).
- **역할 4종**(기존 스키마에 이미 정의됨): 관리자 / 영업담당 / 구매담당 / 생산담당.
- **역할-엔드포인트 매핑**:
  - MDM 생성(품목/고객/공급사) → 관리자
  - 영업(수주 등록/출하/청구) → 영업담당, 관리자
  - 구매(PR/PO 등록/입고처리) → 구매담당, 관리자
  - 생산(생산오더/작업지시/실적입력) → 생산담당, 관리자
  - 승인 결재, 회계 수동전표, AI Agent apply(Buyer/Demand Planner) → 관리자
  - MES/WMS webhook, AI Agent 추천 GET → 이번 범위 밖(기존 설계 그대로 유지)

## 비범위(명시적으로 하지 않음)
- GET 엔드포인트 인증 강제
- JWT/OAuth2, 비밀번호 재설정/이메일 인증 등 실운영 기능
- MES 웹훅 인증(API Key) — 별도 v4 항목으로 유지

## 성공 기준
1. 비밀번호 없이 로그인 시도 시 실패, 올바른 비밀번호로 로그인 시 토큰 발급
2. 발급된 토큰 없이 보호된 POST 호출 시 401
3. 토큰은 있으나 역할이 맞지 않는 POST 호출 시 403 (예: 영업담당이 PO 등록 시도)
4. 올바른 역할로 호출 시 정상 동작(기존 E2E 플로우 회귀 없음)
5. 프론트엔드: 로그인 화면 → 로그인 성공 시 역할에 맞는 메뉴/버튼만 노출, 로그아웃 동작
6. 브라우저 라이브 테스트로 최소 2개 역할(예: 영업담당, 관리자) 로그인 후 동작 차이 확인
