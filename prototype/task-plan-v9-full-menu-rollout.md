## 2. 이번 로드맵의 스코프 원칙

1. **스택 변경 없음** — 기존 FastAPI + SQLAlchemy(SQLite/PostgreSQL 이중지원) + React 19/Vite/
   TypeScript/Tailwind 그대로 확장한다. Spring Boot, Kubernetes, Redis, Vector DB, 마이크로서비스
   분리(3.x/4.x/5.4가 제시하는 Enterprise 아키텍처)는 이번 범위에서 제외한다.
2. **스키마는 additive migration만** — `schema_sqlite.sql`/`schema_postgres.sql`은 계속 수정하지
   않고 `migrations/versions/0004_*` 부터 번호를 이어간다(v6~v8에서 확립한 규칙).
3. **기존 패턴 재사용** — RBAC(`require_roles`), JWT 인증, `write_audit_log`, LOT/Serial 추적,
   AI Agent(rationale+ai_narrative) 등 이미 검증된 패턴을 새 모듈에도 그대로 적용한다. 새 역할이
   필요하면(예: "서비스담당") 최소한으로만 추가한다.
4. **프로토타입 티어로 단순화(Simplified) vs 이번엔 스텁/이연(Deferred)** — 각 웨이브 표에서
   항목별로 구분한다. 예:
   - SPC → 관리도 시각화 없이 Cp/Cpk 수치 계산 API만 (Simplified)
   - 8D Report → 8단계 정식 폼 대신 원인/시정조치 텍스트 필드만 있는 간이 등록 (Simplified)
   - Multi-Company/Multi-GAAP/Multi-Language/연결회계/IFRS/ESG 프레임워크/글로벌 승인 매트릭스 →
     Enterprise 단계 전용이라 이번 로드맵 전체에서 명시적으로 제외(Deferred, 3.8/4.x/5.8 소관)
   - 공급망 시뮬레이션, MRP 최적화 등 "AI/최적화" 성격 항목 → 기존 AI Agent와 동일하게 규칙기반
     간이 로직으로 구현(정교한 솔버 없음)
5. **검증 방식 고정** — 매 웨이브마다: fresh `/tmp` 복사본 pytest 전체 통과 → `npx tsc -b` 통과 →
   eval-report 작성 → README/memory 갱신. 지금까지의 v1~v8과 동일한 패턴.
